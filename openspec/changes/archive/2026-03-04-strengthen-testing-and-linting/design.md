## Context

プロジェクトの初期段階において、Lintの警告やマジックナンバーが残っており、テストカバレッジの強制も行われていません。将来的な拡張に備え、今このタイミングで品質基準を最高レベル（カバレッジ100%、厳格なLintルール、型安全性）に引き上げる必要があります。

## Goals / Non-Goals

**Goals:**
- `pytest-cov` によるカバレッジ100%の強制。
- Ruffの追加ルール（D, ANN, PT, PL, TRY等）の適用。
- 現状のすべてのLint警告（30件以上）の解消。
- マジックナンバーの `Enum` への置き換えによる保守性向上。
- スクリプトを含む全コードのロギング移行。

**Non-Goals:**
- CIパイプラインの構築（別スコープ）。
- 大幅なアーキテクチャの変更。
- 既存のテストケースの動作変更（品質向上のための追加のみ）。

## Decisions

### 1. カバレッジ100%の強制 (`--cov-fail-under=100`)
- **理由**: 「ほぼ100%」は時間とともに低下するため、最初から100%を必須とすることで、テストのないコードの混入を物理的に防ぎます。
- **代替案**: 80%や90%の設定。しかし、小規模なプロジェクトでは100%維持のコストが低く、メリットが上回ると判断。

### 2. Discord Interaction Type の Enum 化
- **理由**: `type == 1` などの記述は可読性が低く、Discord API の仕様変更や理解の妨げになります。
- **代替案**: 定数 (`INTERACTION_TYPE_PING = 1`)。しかし、関連する値をグループ化できる `IntEnum` の方が Pythonic であり、FastAPI との相性も良いため採用。

### 3. テストコードでの Lint 抑制 (`per-file-ignores`)
- **理由**: テストコードでは `assert` やマジックナンバー（期待値としての `200` 等）が本質的に必要です。これらを一律禁止するとテストの記述性が著しく低下します。
- **設定**: `tests/` 配下で `S101` (assert) と `PLR2004` (magic-value) を無視。

### 4. ロギングへの完全移行
- **理由**: `print` は出力先やレベルの制御が困難です。`logging` を使うことで、標準エラー出力や将来的な外部ツール連携が容易になります。

## Risks / Trade-offs

- **[Risk]** カバレッジ100%の維持が負担になる → **[Mitigation]** ボイラープレートを減らし、テストしやすい疎結合な設計を維持する。
- **[Risk]** 厳格なLintルールによる開発速度の低下 → **[Mitigation]** `ruff --fix` を活用し、自動修正可能なものは機械に任せる。
