## Context

現状、コード品質の担保（Lint, Formatter, Type Check, Test）は開発者のローカル環境での実行に依存しています。
GitHub Actions を導入し、プルリクエストおよび `main` ブランチへのプッシュ時に自動的にチェックを実行することで、デグレの防止と品質維持を確実なものにします。
また、将来的な CI/CD の拡張を視野に入れ、役割を明確に分けたワークフロー構成を目指します。

## Goals / Non-Goals

**Goals:**
- プルリクエスト時の自動的な品質チェック（Ruff, Mypy, Pytest）の導入。
- Docker イメージのビルドが常に正常であることを保証するチェックの導入。
- `uv` を活用した高速で再現性のある CI セットアップ。
- 将来的な拡張を考慮したワークフローファイル名の命名。

**Non-Goals:**
- CD（継続的デプロイ）の導入。
- 外部サービス（n8n, Discord API）との実際の通信を含む統合テストの追加（既存のモックテストを CI で実行するにとどめる）。

## Decisions

- **ワークフローファイル名: `python-checks.yaml`**
  - 最初は `ci.yaml` を検討しましたが、将来的にデプロイ用 (`deployment.yaml`) や特定のタスク用ワークフローが追加される可能性が高いため、Python の品質チェックであることを示す明確な名称を採用します。

- **`astral-sh/setup-uv` の採用**
  - プロジェクトで `uv` を使用しているため、GitHub Actions においても公式のアクションを使用して高速な依存関係の解決とキャッシュを行います。

- **Docker ビルドチェックの導入**
  - ソースコードだけでなく、実行環境（Dockerfile）の健全性も同時に検証することで、デプロイ時の予期せぬ失敗を回避します。

## Risks / Trade-offs

- **[Risk] CI 実行時間の増大** → **[Mitigation]** `uv` のキャッシュ機能 (`enable-cache: true`) を活用し、2回目以降のビルド時間を短縮します。
- **[Risk] Docker ビルドの重さ** → **[Mitigation]** 品質チェックとは別のワークフローに分けるか、同一ジョブ内で効率的に実行します。今回はシンプルさのため一つのワークフローに含めますが、将来的に時間がかかる場合はジョブを分割します。

## Migration Plan

1. `.github/workflows/python-checks.yaml` を新規作成してリポジトリにプッシュ。
2. GitHub 上で PR を作成し、CI が正常に実行・完了することを確認。
3. 必要に応じて GitHub のブランチ保護ルールを設定し、CI が成功しないとマージできないように設定する（手動設定）。

## Open Questions

- Docker ビルド時にキャッシュを効かせるための `docker/build-push-action` などの導入要否。
